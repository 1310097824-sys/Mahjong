基于 Python 的雀魂式立直麻将单人对战小游戏实施方
案
执行摘要
本方案面向“单人对战 + 其余全为 AI”的日本立直麻将（包含三麻与四麻）小游戏，目标是在功能与规则层面
尽可能复刻雀魂的规则体验，并以 Python + MySQL 8.0 为核心技术约束，给出可落地的工程化实施路线。规
则层面的关键点包括：四麻与三麻的差异（如三麻去除万子 2–8、无吃、北拔作宝牌等）、途中流局（四风连
打、四杠散了、四家立直、九种九牌）、流局满贯（Nagashi Mangan）、多家荣和与“无三家和了流局”等雀
魂特性，以及立直棒/供托棒与本场棒结算等细节。官方可直接引用的雀魂赛事房间规则参数（例如赤宝牌数
量、食断、供托点数、本场点数、途中流局开关、三响流局开关等）可参考其赛事手册中给出的“Game
rules”配置项。
工程实现上，建议采用“服务端权威牌局内核 + 事件溯源存档”的设计：所有摸打鸣立/翻宝/结算由后端核心
状态机驱动，前端仅提交意图并显示结果；同时将“动作序列（Action Log）+ 随机种子（Seed）+ 关键状态快
照（Snapshot）”落库，实现回放、断线重连、对局审计和 AI 训练数据复用。规则判断与番符计分建议复用成
熟的开源日麻计算库（如 MahjongRepository 的 mahjong），其宣称可计算役种、符、番与点数，并在大量
天凤牌谱上验证过一致性，有助于显著降低“役判/符判/点表”实现风险。
AI 方面，建议按难度分层：从“规则/向听数驱动的启发式 AI”逐步迭代到“策略网络 + 价值网络”的学习型
AI。最强难度如要逼近“联动大教练/高水平教学 AI”的强度，需要引入监督学习与自博弈强化学习的组合路
线：可参考 Suphx 在日麻场景中使用深度强化学习、引入全局回报预测与运行时策略自适应等手段的研究思
路；  三麻方向可参考面向 Sanma 的深度强化学习工作（如 Meowjong），其明确提出三麻与四麻在牌山/
动作集合（Chii 被 Kita 替代）等方面的差异，并采用“多动作模型 + 自博弈强化学习微调”的路径。 高吞
吐模拟训练可参考 Mjx 这类为日麻 AI 研究设计、强调高性能模拟与研究友好 API 的框架思想。
另外，合规层面需要强调：雀魂的文本、美术、音频、角色与客户端资源受著作权与协议约束，开发时应避免
复制其资源与专有内容（可用开源牌面素材或自行绘制），并在产品叙述上避免造成官方授权或同人合作的误
导；同时可参考其用户协议中关于知识产权与打击外挂/未授权程序的条款精神来设计日志、校验与反作弊策
略。

系统架构与技术选型理由
总体架构
建议采用“客户端界面 + 后端牌局服务 + AI 决策服务 + MySQL 持久化”的分层，即便你当下只做单机，也仍
按“本地部署的 C/S”组织代码。这样做的直接收益是：
1) 规则与随机性更容易做到“权威、可复现、可审计”；
2) 回放/重连天然落在后端事件流上；
3) AI 训练与对战跑批可以复用同一套牌局内核，不被 UI 牵制。

以上结构的关键是：把“状态机（合法动作集合）”作为单一真相来源，任何 UI/AI 只是在“当前可行动作集
合”中选择。此模式与赛事房间规则参数的“开关化配置”兼容度高（例如途中流局开关、三响流局开关、食
断/赤宝牌数量等）。
后端与核心引擎
核心引擎建议拆成五个可独立测试的子模块：
State（状态）：完整局面数据结构（手牌、河、鸣牌、副露、立直状态、宝牌指示牌、供托、本场、
剩余牌数、玩家点数等）。
Rules（规则）：从 State 推导合法动作集合，并提供 validate(action, state) 与 
apply(action, state)。
Flow（流程）：实现“谁行动—是否进入响应窗口—如何裁决多家响应—何时结算—何时开新局”的状
态机。
Scoring（计分）：对“和牌形/役/番符/点数/责任支付”进行计算与结算；多家和牌时的供托归属与支
付顺序必须严格一致（雀魂支持多家荣和，且三响不按流局处理）。
Event Sourcing（事件溯源）：每次动作落一条不可变事件（Action Log），可选定期落“快照”加速
重连与回放。
在役判与点数方面，强烈建议复用成熟库：例如 MahjongRepository 的 mahjong（Python）在功能描述中包
含“役种/符/番/得点计算、和牌校验、向听数计算”，并宣称在大量天凤牌谱上验证一致性，可显著降低“计
分实现不一致”的风险。
AI 模块
AI 模块需要同时满足两类需求：
1) 在线对局实时决策（低延迟、稳定、可解释）
2) 离线训练/评估（高吞吐、可并行、可回放对照）
因此建议 AI 模块提供两套接口：
act(observation, legal_actions, budget_ms) -> action：对局实时决策
selfplay(n_games, ruleset, seed) -> replays：自动生成训练数据与评估数据
当进入“最强教练级”阶段，建议引入深度学习框架与训练库：Tianshou 作为 PyTorch DRL 库有较完整算法
与统一接口，并在 JMLR 有论文介绍其模块化与训练基础设施特点。
数据库与数据结构策略
MySQL 8.0 主要用于：用户数据、对局结果、动作日志、AI 配置版本、排行榜与统计聚合。对“规则配置/动作
payload/快照元数据”等结构化但频繁演进的字段，建议使用 JSON 列；如需对 JSON 内的关键字段做索引，
可用 生成列（generated column）+ 索引 的方式实现间接索引，这在 MySQL 官方文档中属于推荐范式。

前端/界面
你一个人开发，界面建议优先“信息清晰 + 操作顺畅”，而非高质量 3D/粒子特效。可选两条路线：
桌面客户端（推荐单人开发）：Qt（PySide6）或类似 GUI，优势是部署简单、无需浏览器适配、与本
地后端同机运行。
Web 前端：浏览器 UI + WebSocket，优势是跨平台与后期扩展；但会引入更多前端工程量。
无论选哪条，都建议 UI 与规则引擎之间只通过“状态快照 + 合法动作列表”交互，避免 UI 内产生任何“私自
推演规则”的分叉逻辑。
部署形态
本地开发/运行：Docker Compose（后端 + MySQL）+ 本地 UI。 
云端可选：单人对战若考虑“云端排行榜/牌谱分享”，则后端与数据库上云，客户端只做 UI；但要接
受“单人也要维护服务”的运维成本。 
MySQL 8.0 数据库表设计
设计原则
1) 事件溯源优先：动作日志是回放与审计的根；局面快照是优化手段而非真相来源。
2) 可复现：每局/每小局记录随机种子与规则版本，保证“同一动作序列可复盘到同一结果”。
3) AI 可追责：AI 的每次决策应记录“模型版本/参数/随机温度/搜索深度”等，便于回归与对比评测。
4) 统计可聚合：排行榜与统计用“汇总表 + 物化更新”或“定时任务重算”均可，但必须能从原始对局数据重
建。

L1-L3（不依赖机器学习的可交付路线）
1) 向听数与进张计算
需要快速计算：shanten(hand)、ukeire(hand)、waits(hand)。开源 mahjong 库包含向听与和牌
校验能力，可用于快速落地。
2) 攻守权衡（启发式）
进攻得分近似：EV_attack ≈ P(win) * expected_points - P(deal_in) * expected_loss。 
防守危险度估计：采用经验规则（现物 0%、筋、壁、字牌危险度随巡目变化）+ 立直后危险度上调。
3) 三麻特化
三麻手进展更快、打点更大、且存在 North/Kita 动作（会打断一发/两立直等），策略需更激进；
Meowjong 也指出三麻相较四麻更偏进攻风格。
三麻得点需要支持自摸损与北折半两种结算机制（默认 Ranked 为自摸损，友人场可北折半）。

L4-L5（学习型/教练级路线）
1) 环境与模拟吞吐
强化学习对样本量要求极高，因此需要高性能模拟器或尽可能快的自研内核。Mjx 论文强调其作为开源日麻框
架，核心目标之一是提升模拟速度，并报告其在基准中可达到数量级的加速。
2) 训练数据来源
监督学习（SL）：可用公开牌谱（需注意许可/协议）或自生成牌谱（用较强启发式 AI 作为
teacher）。 
自博弈强化学习（RL）：从 SL 初始化出发，自博弈产生轨迹并用 PPO/PG 等更新策略。Meowjong 的
流程是“多动作 CNN 预训练 + discard 模型自博弈 RL 增强”。
3) 评估指标
局级：和率、自摸率、放铳率、立直率、副露率、平均打点、流局听牌率。 
场级：平均顺位、1 位率、pt/elo、不同点差局面的策略合理性（All Last 的抉择）。 
教练级还要加：与“基准 AI”对战的统计显著性、以及“推荐手一致率/后验价值提升”。
可复用算法与模块清单
向听/有效牌（ukeire）计算器：L1-L5 全用。 
危险度模型（Defence model）：L2 起用；L3-L5 逐步替换为学习模型。 
期望值估计器（EV estimator）：L3-L5 全用。 
搜索器（rollout / IS-MCTS）：L4；L5可作为推理时增强（类似运行时策略自适应的“在线修正”思
想）。
性能/延迟预算
参考雀魂赛事房对局计时“Thinking Time: 5+20s”这类在线时间配置，可以把在线 AI 推理时间预算设为： 
L1-L3：50–150ms/决策（体验顺滑） 
L4：200–800ms/决策（可接受轻微思考） 
L5：100–300ms（神经网络推理）+ 可选 200–1000ms（轻量搜索）
赛事房规则截图可作为“目标体验”的参考基线。
单人对战支撑能力：匹配、回放、存档、重连、反作弊、日志调试
单人匹配与开局编排
单人对战本质是“开房间自动补 AI”： 
输入：模式（3P/4P）、时长（东风/半庄）、AI 难度分配（例如 1 个困难 + 2 个普通）、规则预设。 
输出：games、game_players、首局 rounds 初始化。 
建议做一个“编排器（match orchestrator）”层：它不懂规则细节，只负责选择规则集与 AI 配置、生成种子
并启动引擎。

回放与牌谱
核心设计：Action Log=唯一真相
每个动作写入 actions（seq 严格递增），并存 state_hash_pre/post。 
回放时从 RoundInit 开始重放动作序列即可复原全局。 
关键点：所有“系统动作”也必须记录（发牌、翻宝牌、流局结算），避免回放时依赖隐含逻辑。 
导出格式建议
JSON Lines（每行一个 action）+ 头部 metadata（ruleset/seed/版本）。 
若后期想做 AI 训练，可直接把回放转成“(observation, legal_actions, chosen_action, reward)”样
本。
存档与断线重连
断线重连对单机也有意义（崩溃恢复/意外退出）：
每 N 条 action（例如 20 条）落一个 state_snapshots：压缩后的完整状态。 
重连：加载最后快照 + 重放其后的 action。 
校验：重放后 state_hash_post 必须一致，否则判为存档损坏并触发“回滚到更早快照/只读回放模
式”。
作弊防护与审计思路
单人本地对战的作弊动机主要来自“刷排行榜/刷战绩”。完全离线无法绝对防作弊，但可以做到：篡改成本显
著上升 + 审计可发现。
建议策略：
1) 服务端权威
不信任客户端：客户端只发“我想打这张牌/我想立直”，后端验证后才落 action。
2) 动作链哈希（hash chaining）
state_hash_post = H(state_hash_pre + action_payload + canonical_state)，形成不可轻易伪造
的链。
3) 随机性承诺（commit-reveal 可选）
开局先记录 seed_commit = H(seed)；终局再把 seed 明文写入结果，防止“看到牌山后改 seed”。
4) 异常检测
对 AI/玩家决策耗时、非法动作次数、重连频率做统计，触发告警。
5) 设计对齐行业反作弊原则
雀魂协议中提到会用技术手段识别未授权、危害公平的程序，并可能收集与此相关的信息；虽然你不需
要复刻其做法，但可以借鉴“对局日志 + 异常检测 + 处罚/标记”的治理思路。

L5：需要搭建训练流水线、数据、分布式对局跑批与评测；参考 Suphx/Meowjong/Mjx 的研究路线意
味着投入会显著上升。
测试计划
测试应覆盖三个维度：规则正确性、数据一致性、AI 回归稳定性。
1) 单元测试（Unit）
役判测试：每个役至少 5–20 个牌例（含开门/门清、里宝、一发、岭上等边界）。 
符测试：双风雀头 4 符、平和自摸/荣和符、坎张/边张/单骑等。
流局测试：九种九牌触发条件、四家立直触发条件、三麻不出现四风连打/四家立直。
2) 集成测试（Integration）
“固定 seed + 固定 AI”跑 1000 局，断言结果可复现（action hash 链一致）。 
“随机 seed + 固定 AI”跑 1 万局，断言不崩溃、牌数守恒、无非法状态。
3) AI 对战回归测试（Regression）
每个 AI 版本在同一组 seed 上对打 N 局，比较关键指标（放铳率/平均顺位/平均打点）是否出现非预期
劣化。 
对 L5：增加“策略漂移”监控（模型更新后与旧模型对局胜率需在合理区间）。
