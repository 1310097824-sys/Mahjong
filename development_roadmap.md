# 后续发展路线图：朝雀魂式立直麻将体验推进

更新时间：2026-05-07

本文档用于规划当前麻将系统接下来的详细发展过程。目标不是直接复制雀魂的素材、角色、商标或专有资源，而是把系统体验推进到“像成熟网麻产品一样完整、顺滑、稳定、可复盘、可成长”的方向。

当前系统已经具备浏览器牌桌、MySQL 持久化、四麻/三麻、基础雀魂规则对齐、AI L1/L2/L3、行动提示、历史对局、回放和文档体系。接下来应该从“能玩”继续推进到“耐玩、稳定、漂亮、规则严谨、AI 有网麻味道”。

## 1. 总体目标

最终希望形成一套本地优先、可逐步扩展到联机的立直麻将系统。

核心体验目标：

- 规则上接近雀魂/标准立直麻将，四麻和三麻都能稳定运行。
- 牌桌上接近成熟网麻体验，信息清楚、动画顺滑、操作不挡视线。
- AI 不只是会随机打牌，而是有明确攻防判断、押退逻辑和局况意识。
- 行动提示能成为“牌理教练”，让玩家知道为什么推荐这一步。
- 回放和统计能支持复盘、查错、分析自己和 AI 的打法。
- 数据结构能支撑长期历史、规则审计和未来 AI 训练。
- 前端保持原创视觉，不直接使用雀魂专有素材。

## 2. 发展原则

### 2.1 规则优先

麻将系统的底层体验首先取决于规则正确性。任何 UI 和 AI 改动都不能破坏规则推进。

后续每补一个规则点，都应该同步补：

- 后端合法动作生成。
- 后端执行逻辑。
- 结算逻辑。
- 前端显示。
- 自动化审计或单元测试。
- `logic.md` 文档说明。

### 2.2 后端是唯一规则源

前端只负责展示和提交动作，不应该在前端裁定能不能吃碰杠、能不能和牌、能不能立直。

后端继续保持唯一规则源：

- 合法动作由 `app/engine.py` 生成。
- 前端只展示 `legal_actions`。
- 前端提交 `action_id`。
- 后端执行前再次校验动作合法性。

### 2.3 UI 像网麻，但素材原创

可以学习成熟网麻的信息组织方式，例如：

- 中心桌芯。
- 四家方位灯。
- 牌河中心矩阵。
- 立直横置牌。
- 副露方向展示。
- 结算分层。
- 牌谱回放控制。
- Dock 式功能面板。

但不要直接复制雀魂角色、背景、音效、LOGO、UI 图标或专有美术资产。

### 2.4 AI 先可解释，再追求强度

当前 AI 是规则型/EV 型。短期不建议直接上深度学习，因为成本高、调试难、训练数据不足。

更适合当前项目的路线是：

1. 规则型 EV 继续打磨。
2. 加强对手建模。
3. 加强押退判断。
4. 做更可靠的浅层搜索。
5. 做牌谱回放评测。
6. 最后再考虑模型训练或外部 AI 对照。

### 2.5 每个阶段都要可验证

每个版本完成后都应该至少跑：

```powershell
python -m py_compile app\engine.py app\main.py app\store.py tests\mahjong_soul_rule_audit.py
python tests\mahjong_soul_rule_audit.py
cd riichi-mahjong-ui
npm run lint
npm run build
```

如果涉及性能，还应补充浏览器性能观察和对局压力测试。

## 3. 阶段 0：当前 v2.0 基线

当前系统已经完成：

- React 浏览器牌桌。
- MySQL 持久化。
- 四麻/三麻。
- 东风/半庄。
- 段位默认/友人场/古役房。
- 吃、碰、杠、立直、自摸、荣和、拔北、九种九牌。
- 中心桌芯和中心牌河。
- 传统麻将牌面。
- Dock 面板。
- 可拖动行动提示小窗。
- 规则审计脚本。
- 文档体系。

当前主要短板：

- 规则虽然已经覆盖很多点，但还需要更多边界用例。
- AI 还不是强 AI，更多是规则型 EV 和浅层搜索。
- 回放还偏“看过程”，缺少复盘分析。
- 数据库仍是一张 `games` 表加 JSON，长期统计能力有限。
- 前端已经接近网麻牌桌，但动画、音效、节奏和结算表现还能继续精修。
- 还没有账号、段位、任务、角色、装扮、联机和匹配系统。

## 4. 阶段 1：v2.1 规则稳定化

目标：把规则从“基本能对齐”推进到“边界情况也可审计”。

### 4.1 重点任务

- 扩展 `tests/mahjong_soul_rule_audit.py`，把每个复杂规则拆成更细用例。
- 增加立直后自动摸切、立直后暗杠、抢杠、岭上、海底、河底的组合测试。
- 增加三麻拔北后补牌、拔北被抢、拔北宝牌计数测试。
- 增加多响供托、本场归属、庄家连庄、终局条件测试。
- 增加古役开关测试，确保段位场不会误开古役。
- 增加三麻移除 `2-8 万` 后的进张和宝牌指示测试。
- 增加异常动作测试，例如重复提交、旧 action_id、非法打牌、非当前玩家动作。

### 4.2 验收标准

- 规则审计至少覆盖 50 个以上 PASS 项。
- 常见边界规则都有明确测试名。
- 前端无法通过伪造 action_id 绕过后端规则。
- 每次修改规则都能快速知道有没有破坏旧逻辑。

## 5. 阶段 2：v2.2 牌桌体验精修

目标：让牌桌更像成熟网麻产品，操作更舒服，信息更清楚。

### 5.1 牌桌布局

- 继续压缩和精修中心桌芯，让它更像小型电子桌芯。
- 三麻桌芯只显示三家方向灯，四麻桌芯显示四家方向灯。
- 牌河固定中心矩阵，不随着出牌数量漂移。
- 各家手牌贴近桌边，牌河靠近中心，副露在各家手牌右下角且不堆叠。
- 当前玩家操作按钮不要遮挡手牌、牌河和副露。

### 5.2 动画节奏

- 摸牌、出牌、吃碰杠、立直、和牌做成统一节奏。
- AI 出牌延迟继续保留 1 秒 / 2 秒 / 3 秒 / 关闭动画选项。
- 立直宣言、横置牌、供托棒、和牌闪光做成更明确的反馈。
- 结算面板支持滚动，重要信息优先显示。

### 5.3 音效与反馈

- 加入原创音效，不使用雀魂素材。
- 动作音效分层：摸牌、出牌、吃、碰、杠、立直、荣和、自摸、结算。
- 支持音量开关和静音。
- 支持低动效模式，避免低性能设备卡顿。

### 5.4 验收标准

- 桌面端 1920x1080 下牌桌比例自然。
- 1366x768 下也能完整操作，不需要频繁滚动。
- 三麻和四麻布局都不遮挡核心信息。
- 行动提示小窗、Dock 面板、结算面板互不遮挡。

## 6. 阶段 3：v2.3 AI 强化第一轮

目标：让 AI 从“会打”变成“有攻防感”。

### 6.1 弃牌 EV 改进

继续强化 `discard_profile()`：

- 更细分速度 EV、打点 EV、防守 EV、局况 EV。
- 对听牌、一向听、两向听分别使用不同权重。
- 对门清、断幺、役牌、染手、七对、对对、混一、清一等路线给更稳的判断。
- 对宝牌、红宝牌、拔北宝牌的价值做更准确估算。

### 6.2 对手建模

强化 `opponent_models()`：

- 判断对手是否立直。
- 判断对手是否副露快攻。
- 判断对手是否染手。
- 判断对手是否役牌路线。
- 判断对手是否对对路线。
- 估算对手打点区间。
- 根据巡目提高危险权重。
- 根据自己点差决定押退倾向。

### 6.3 鸣牌判断

继续优化 `open_call_profile()`：

- 避免无役副露。
- 避免远向听乱碰。
- 明确役牌、断幺、染手、对对、速攻、追分场景。
- 开杠要更谨慎，考虑给别人新宝牌的风险。
- L1 更容易乱鸣，L2 有基本役种意识，L3 强调速度、役种和风险综合。

### 6.4 验收标准

- AI 不再频繁为了碰牌破坏和牌路线。
- L3 面对立直时会明显更愿意打现物或低风险牌。
- L3 在落后时会更愿意进攻，领先末局会更愿意守。
- 行动提示能解释为什么推荐鸣或不鸣。

## 7. 阶段 4：v2.4 AI 强化第二轮

目标：让 AI 开始具备 AlphaJong 式“模拟几步后再选”的味道。

### 7.1 浅层搜索

强化 `alpha_style_lookahead_profile()`：

- 对每个候选弃牌模拟未来 1-2 巡。
- 只保留最有价值的摸牌分支。
- 对每个分支重新计算最佳弃牌。
- 用剩余张数加权未来收益。
- 对搜索结果做缓存，减少卡顿。

### 7.2 蒙特卡洛雏形

不直接上大规模训练，先做轻量模拟：

- 根据可见牌生成未知牌池。
- 对候选动作随机模拟若干次。
- 估计听牌率、和牌率、放铳率。
- 只在 L3 或行动提示深度分析时启用。

### 7.3 AI 评测体系

建立 AI 回归评测：

- 固定随机种子跑 100 局。
- 记录 L1/L2/L3 平均顺位。
- 记录副露率、立直率、放铳率、和牌率。
- 记录流局听牌率。
- 记录 AI 每局平均耗时。

### 7.4 验收标准

- L3 长期表现稳定强于 L2，L2 稳定强于 L1。
- AI 决策耗时不会明显卡住前端。
- 行动提示和 AI 实际动作基本一致。

## 8. 阶段 5：v2.5 回放、统计和牌谱分析

目标：让系统从“打一局”升级成“能复盘、能分析、能成长”。

### 8.1 回放增强

- 回放支持逐手、逐巡、自动播放、倍速播放。
- 支持跳到立直、鸣牌、和牌、放铳、流局等关键节点。
- 回放时展示当时合法动作和 AI 推荐。
- 回放结束展示整局摘要。

### 8.2 统计增强

- 玩家统计增加局数、平均顺位、一位率、二位率、三位率、四位率。
- 增加和牌率、放铳率、副露率、立直率、自摸率、流局听牌率。
- 增加三麻和四麻分开统计。
- 增加东风和半庄分开统计。
- 增加不同规则档位分开统计。

### 8.3 牌谱导入导出

- 支持导出本项目 JSON 牌谱。
- 支持导入本项目 JSON 牌谱。
- 长期可以考虑兼容标准牌谱格式或 mjai 风格日志。

### 8.4 验收标准

- 玩家能用系统回看自己为什么输。
- 每局关键节点可快速定位。
- 统计不再只显示总局数和平均顺位。

## 9. 阶段 6：v2.6 数据库结构升级

目标：从“一张 games 表 + JSON”升级到更适合长期统计和 AI 训练的数据结构。

### 9.1 拆表方向

建议逐步拆成：

| 表名 | 作用 |
| --- | --- |
| `games` | 对局基础信息和当前摘要 |
| `game_players` | 每局每个玩家的座位、名称、AI 等级、最终点数和顺位 |
| `game_actions` | 每一步动作日志 |
| `game_snapshots` | 回放快照 |
| `game_results` | 结算总览 |
| `game_result_yaku` | 每次和牌的役种、番数、符数 |
| `player_stats` | 预聚合玩家统计 |
| `ai_decision_logs` | AI 候选动作、EV、风险、最终选择 |

### 9.2 迁移工具

- 引入 Alembic 管理数据库迁移。
- 保留旧 `games` JSON 数据读取兼容。
- 写迁移脚本把旧 JSON 拆到新表。
- 所有迁移可重复执行、可回滚。

### 9.3 验收标准

- 旧对局数据不丢。
- 新旧版本数据可兼容读取。
- 统计查询速度明显提升。
- AI 决策日志可以被单独分析。

## 10. 阶段 7：v3.0 账号、段位和长期游玩

目标：让系统有长期成长感，而不是每次只打一局。

### 10.1 玩家账号

- 本地账号系统。
- 玩家昵称、头像、偏好设置。
- 规则偏好、动画偏好、音量偏好。
- 最近使用配置自动保存。

### 10.2 段位系统

可以做原创段位体系，不直接照搬雀魂名称。

建议包含：

- 段位分。
- 升降级。
- 四麻和三麻分开段位。
- 东风和半庄分开记录。
- AI 对战和真人对战分开记录。

### 10.3 任务和成就

- 首次立直。
- 首次满贯。
- 首次役满。
- 连续三局不放铳。
- 三麻拔北自摸。
- 古役房特殊役完成。

### 10.4 验收标准

- 玩家能看到长期成长轨迹。
- 系统能保存个人偏好。
- 段位和统计不会被测试局污染，可以区分正式局和练习局。

## 11. 阶段 8：v3.5 联机和实时对战

目标：从本地单人对战扩展到多人实时对战。

这一阶段工作量很大，应该在规则、AI、回放和数据库都稳定后再做。

### 11.1 WebSocket 实时通信

- 使用 WebSocket 推送牌局状态。
- 每个玩家只收到自己可见的信息。
- 后端继续作为唯一规则源。
- 支持断线重连。
- 支持超时自动摸切。

### 11.2 房间系统

- 创建房间。
- 加入房间。
- 房间规则配置。
- 准备状态。
- 观战模式。
- 私密房间邀请码。

### 11.3 匹配系统

- 本地简单匹配队列。
- 按模式匹配：四麻、三麻、东风、半庄。
- 长期可按段位分匹配。

### 11.4 反作弊和状态安全

- 前端永远不能知道他家真实手牌。
- 所有动作必须由后端校验。
- 状态推送必须做可见性过滤。
- 关键动作要记录日志。

### 11.5 验收标准

- 两个浏览器可以实时打一局。
- 刷新页面后能回到当前房间。
- 断线重连后状态正确。
- 他家手牌不会泄露给前端。

## 12. 阶段 9：v4.0 高级 AI 和教练系统

目标：让系统不只是陪打，而是能帮助玩家变强。

### 12.1 AI 训练和评测数据

- 保存 AI 每次候选动作。
- 保存每个候选动作的 EV 组成。
- 保存最终结果。
- 建立“推荐动作是否导致更好结果”的长期评测。

### 12.2 教练模式

- 玩家打出一张牌后，系统判断是否偏离前三推荐。
- 给出简短解释。
- 回放时标出关键失误。
- 支持只看结论和看详细分析两种模式。

### 12.3 外部 AI 对照

长期可以研究：

- AlphaJong 思想对照。
- Suphx 论文思路对照。
- NAGA/Mortal 类工具的理念参考。
- 自建模型或轻量价值网络。

### 12.4 验收标准

- 玩家能在对局后看到 3-5 个关键改进点。
- 行动提示不只是推荐，还能解释牌理。
- AI 决策日志可以用于后续训练数据。

## 13. v4.0 之后：深度学习 AI 专项路线

目标：在 v4.0 的高级 AI 教练和决策日志基础上，继续发展出可以使用大量数据集训练、评测、部署和持续迭代的深度学习麻将 AI。

这一阶段是一个新的子项目级工程，不只是“把模型接进来”。它至少包含数据采集、数据清洗、特征编码、动作空间、监督学习、价值网络、强化学习、自对弈、模型评测、推理服务、模型版本管理和前端解释展示。

### 13.1 总体目标

深度学习 AI 的最终目标不是让系统黑箱化，而是让系统同时具备：

- 更强的牌效率。
- 更接近高水平网麻的押退判断。
- 更准确的鸣牌、立直、默听、弃和选择。
- 对点差、亲家、终局条件的局况理解。
- 可复盘、可解释、可量化评测的模型输出。
- 可持续训练的数据闭环。

建议长期形成三层 AI：

| 层级 | 作用 | 说明 |
| --- | --- | --- |
| 规则引擎 | 保证合法性 | 所有动作仍由后端规则引擎生成合法动作集合 |
| 神经网络模型 | 评估局面和动作 | 输出动作概率、局面价值、风险、和牌率、放铳率等 |
| 搜索/决策器 | 最终选择动作 | 结合合法动作、模型输出、局况规则和必要搜索做最终决策 |

也就是说，即使有深度学习模型，前端和模型都不能越过 `app/engine.py` 直接执行非法动作。

### 13.2 发展边界

必须明确几条边界：

- 不直接抓取、破解或滥用雀魂客户端数据。
- 不使用没有授权的牌谱数据集。
- 不复制雀魂专有 AI、角色、美术、音效或 UI 素材。
- 不把模型训练结果伪装成 Suphx、NAGA、Mortal 或其他第三方 AI。
- 模型输出只是建议，最终动作仍需通过本项目规则引擎校验。

推荐数据来源优先级：

1. 本系统自产生的对局数据。
2. 本系统 AI 自对弈数据。
3. 玩家明确同意保存的本地牌谱数据。
4. 开源且许可清晰的麻将 AI 数据或框架。
5. 只在合规前提下研究公开牌谱格式。

### 13.3 前置条件

在开始深度学习前，必须先完成这些工程基础：

- 规则审计足够稳定，避免用错误规则生成训练数据。
- 对局状态可完全序列化和重放。
- 每一步动作都有明确 `action_id`、合法动作集合和最终选择。
- 回放能还原当时视角，不能泄露他家暗手牌。
- 数据库能保存大量动作级日志。
- AI 决策耗时可统计。
- 随机种子可复现。
- 三麻和四麻的规则差异可以在样本中明确编码。

如果规则引擎仍频繁卡局，或者某些边界规则不稳定，不建议马上训练大模型。否则模型会学习到错误状态。

### 13.4 数据库和数据仓库升级

深度学习训练不能继续只依赖单行 `games.state_json`。需要把动作级数据拆出来。

建议新增表：

| 表名 | 作用 |
| --- | --- |
| `training_games` | 每局训练对局的元信息，如规则、模式、种子、来源、版本 |
| `training_players` | 每局每家的身份、AI 版本、最终点数、顺位 |
| `training_steps` | 每一步决策样本，包含状态、合法动作、实际动作、奖励 |
| `training_features` | 可选，保存预处理后的特征张量路径或哈希 |
| `training_datasets` | 数据集版本、来源、样本数、生成脚本、校验哈希 |
| `model_versions` | 模型版本、训练参数、指标、文件路径、是否上线 |
| `model_evaluations` | 模型对战结果、规则审计、性能指标、回归测试 |
| `ai_inference_logs` | 线上推理输入摘要、输出、耗时、最终动作 |

训练大数据不建议全部塞进 MySQL。更合理的结构是：

- MySQL 保存索引、元数据、统计结果和文件路径。
- 本地磁盘或对象存储保存 `.npz`、`.parquet`、`.pt`、`.safetensors`、`.onnx` 等大文件。
- 每个数据集版本都有 manifest 文件，记录样本数量、特征版本、动作空间版本和生成脚本版本。

### 13.5 数据样本格式

每一个训练样本应该对应“某个玩家在某一时刻需要做决策”。

建议样本包含：

| 字段 | 说明 |
| --- | --- |
| `sample_id` | 样本唯一 ID |
| `game_id` | 对局 ID |
| `step_no` | 第几步动作 |
| `seat` | 决策者座位 |
| `mode` | `3P` 或 `4P` |
| `rule_profile` | 段位、友人、古役房等 |
| `phase` | 自家回合或响应阶段 |
| `observation` | 当前玩家可见状态 |
| `legal_action_mask` | 合法动作掩码 |
| `chosen_action` | 实际选择动作 |
| `teacher_action` | 规则型 AI 推荐动作，可选 |
| `final_rank` | 终局顺位 |
| `final_score` | 终局点数 |
| `round_delta` | 本局点数变化 |
| `win_flag` | 是否和牌 |
| `deal_in_flag` | 是否放铳 |
| `tenpai_flag` | 流局是否听牌 |
| `reward` | 训练用奖励 |

注意：`observation` 必须只包含该玩家当时能看到的信息。不能把其他玩家暗手牌作为普通输入，否则模型会变成作弊 AI。

### 13.6 特征编码

麻将是非完美信息游戏，特征编码必须非常严谨。

建议先用多通道张量，不要一开始就上过于复杂的 Transformer。

基础特征：

- 自家手牌 34 种牌计数。
- 自家副露。
- 各家牌河。
- 各家立直状态。
- 各家副露数量。
- 宝牌指示牌。
- 赤宝牌配置。
- 三麻拔北数量。
- 当前场风。
- 自风。
- 本场。
- 供托。
- 剩余牌数。
- 巡目。
- 当前分数。
- 当前顺位。
- 点差。
- 亲家位置。
- 是否东风 / 半庄。
- 是否三麻。
- 是否古役房。

风险和牌理辅助特征：

- 当前向听。
- 有效进张数。
- 当前听牌等待。
- 现物集合。
- 筋候选。
- 壁候选。
- 对手威胁等级。
- 对手副露路线推测。
- 当前规则型 AI 的 EV 分解。

是否保留规则型 AI 特征要分阶段：

- 早期可以保留，帮助模型快速学到可解释启发。
- 中后期要做消融实验，比较“有规则特征”和“纯状态特征”的强度差异。

### 13.7 动作空间设计

动作空间必须统一，否则模型无法同时处理弃牌、吃、碰、杠、立直、自摸、荣和、拔北和过。

建议动作空间分成两层：

1. 动作类型。
2. 动作参数。

动作类型包括：

- `discard`
- `riichi_discard`
- `tsumo`
- `ron`
- `pass`
- `chi`
- `pon`
- `open_kan`
- `closed_kan`
- `added_kan`
- `kita`
- `abortive_draw`

动作参数包括：

- 弃牌的 34 种牌型。
- 是否红五。
- 吃牌组合。
- 碰/杠对应牌。
- 具体后续打出哪张牌。

模型输出不要直接使用前端 `action_id`，因为 `action_id` 是运行时生成的。更稳定的方式是：

- 模型输出标准动作编码。
- 后端把标准动作编码映射到当前合法 `ActionChoice`。
- 如果映射不到，则按合法动作 mask 排除。

### 13.8 奖励设计

深度学习麻将 AI 最难的是奖励设计。

建议分阶段使用：

| 阶段 | 奖励 | 用途 |
| --- | --- | --- |
| 监督学习 | 模仿目标动作 | 学会基本牌效率和操作习惯 |
| 局内价值 | 本局点数变化 | 学会和牌、放铳、流局收益 |
| 半庄价值 | 终局顺位和最终点数 | 学会局况、守位、追分 |
| 辅助奖励 | 听牌率、和牌率、放铳率 | 稳定训练和解释输出 |

建议不要只用“是否和牌”作为奖励，因为麻将里很多正确选择是防守、弃和或保顺位。

更好的奖励组合：

```text
reward =
  终局顺位奖励
  + 最终点数变化
  + 本局收益
  + 和牌奖励
  - 放铳惩罚
  + 流局听牌奖励
  - 非法动作惩罚
  - 超时惩罚
```

其中终局顺位奖励可以按模式区分：

- 四麻：一位明显正奖励，四位明显负奖励。
- 三麻：一位正奖励，三位负奖励，二位接近中性。

### 13.10 v4.2：监督学习 baseline

目标：训练第一个“能模仿当前规则型 AI”的策略模型。

推荐技术：

- PyTorch 作为训练框架。
- TensorBoard 记录 loss、accuracy、top-k accuracy。
- 数据集先用本系统规则型 AI 生成。

模型输入：

- 状态特征张量。
- 合法动作 mask。

模型输出：

- 每个合法动作的概率。

训练目标：

- 交叉熵损失。
- 只在合法动作上计算概率。
- 记录 top-1、top-3、top-5 命中率。

目录建议：

```text
ml/
├─ datasets/
├─ models/
├─ training/
├─ evaluation/
├─ inference/
└─ README.md
```

验收标准：

- 模型能在验证集上达到较高 top-3 命中率。
- 模型推理可以返回合法动作概率。
- 模型不会推荐非法动作。
- 离线推理单次耗时可控。

### 13.11 v4.3：价值网络和风险网络

目标：模型不只模仿动作，还要能评估局面好坏。

新增输出头：

| 输出头 | 目标 |
| --- | --- |
| `policy_head` | 推荐动作概率 |
| `rank_value_head` | 预测终局顺位收益 |
| `score_delta_head` | 预测最终点数变化 |
| `win_prob_head` | 预测本局和牌概率 |
| `deal_in_prob_head` | 预测放铳概率 |
| `tenpai_prob_head` | 预测流局听牌概率 |

训练方式：

- 使用多任务学习。
- 动作模仿使用交叉熵。
- 顺位、点数变化使用回归损失。
- 和牌、放铳、听牌使用二分类损失。
- 不同 loss 需要动态或手动权重。

验收标准：

- 价值网络能区分明显好局面和坏局面。
- 放铳风险预测能和规则型风险评估大致相关。
- 行动提示可以展示模型预测的和牌率、放铳率、顺位收益。

### 13.12 v4.4：模型接入行动提示

目标：让深度学习模型先作为“顾问”，不要马上替代 AI。

接入方式：

- 后端新增 `ai/model_service.py`。
- 当前规则型 AI 继续生成候选动作和 EV。
- 模型对同一批候选动作输出概率和价值。
- 行动提示展示“规则推荐”和“模型推荐”。
- 如果两者不同，展示差异原因。

前端展示：

- 首选动作。
- 模型置信度。
- 预测和牌率。
- 预测放铳率。
- 预测顺位收益。
- 与规则型 AI 的分歧。

验收标准：

- 模型可在本地 CPU 推理。
- 推理失败时自动回退规则型 AI。
- 前端不会因为模型服务异常卡死。
- 玩家可以选择关闭模型提示。

### 13.13 v4.5：离线强化学习

目标：利用已有牌谱和自对弈日志，让模型不只是模仿当前 AI。

可尝试方向：

- Conservative Q-Learning 类思想。
- Advantage-weighted behavior cloning。
- 从终局奖励反推动作价值。
- 使用规则型 AI 作为行为策略起点。

注意事项：

- 离线强化学习很容易学偏。
- 必须先做好评测环境。
- 不能只看训练 loss，要看对战结果。
- 每个模型必须和固定基准 AI 对战足够多局。

验收标准：

- 离线 RL 模型在固定评测中强于纯监督 baseline。
- 放铳率没有异常升高。
- 决策不会明显变得极端，例如疯狂副露或过度弃和。

### 13.14 v4.6：自对弈训练

目标：让模型通过大量自对弈继续强化。

自对弈系统需要：

- 无 UI 批量对局 runner。
- 多进程或多机器并行。
- 固定随机种子和可复现日志。
- 模型池。
- 对手采样策略。
- 训练数据自动写入数据集版本。

自对弈对手组合：

- 当前最新模型。
- 历史模型。
- 规则型 L1/L2/L3。
- 随机弱 AI。
- 专门防守型 AI。
- 专门速攻型 AI。

训练流程：

1. 当前模型批量自对弈。
2. 写入新样本。
3. 训练候选模型。
4. 候选模型进入评测。
5. 超过基准后加入模型池。
6. 不达标则丢弃或保留为实验模型。

验收标准：

- 能稳定无人值守跑 10,000 局以上。
- 每个模型都有清晰胜率、平均顺位、放铳率、和牌率。
- 新模型必须通过回归测试才能上线。

### 13.15 v4.7：搜索结合神经网络

目标：让模型不只是单步输出，而是参与搜索。

可行方案：

- 模型给候选动作排序。
- 搜索只展开前 N 个动作。
- 价值网络评估搜索叶子节点。
- 对关键局面做更深搜索。
- 对普通局面只做快速推理。

不建议一开始做完整 MCTS，因为麻将隐藏信息多、动作空间复杂、模拟成本高。

更现实的路线：

```text
合法动作生成
→ 模型策略排序
→ 取前 N 个动作
→ 模拟 1-2 巡
→ 价值网络评估
→ 结合风险和局况修正
→ 输出最终动作
```

验收标准：

- 搜索模型强于单步模型。
- 推理耗时仍然适合本地游玩。
- 用户可以在设置中选择快速模式或深度模式。

### 13.16 v4.8：模型服务化

目标：把模型从后端规则代码里拆出来，形成可替换服务。

推荐两种部署模式：

| 模式 | 说明 |
| --- | --- |
| 内嵌模式 | FastAPI 进程内直接加载 PyTorch / ONNX 模型 |
| 独立服务 | 单独启动 `ml_inference_server`，后端通过 HTTP 或本地 RPC 调用 |

本地项目初期推荐内嵌模式，后期推荐独立服务。

模型格式：

- 训练阶段使用 PyTorch `.pt` 或 `.safetensors`。
- 推理阶段可导出 ONNX。
- CPU 推理优先考虑 ONNX Runtime。
- GPU 推理可保留 PyTorch 或 ONNX Runtime GPU。

推理接口建议：

```text
POST /api/model/predict-action
POST /api/model/evaluate-state
GET  /api/model/version
POST /api/model/reload
```

验收标准：

- 模型版本可切换。
- 推理超时自动回退。
- CPU 模式可用。
- GPU 模式可选。
- 推理日志能记录模型版本和耗时。

### 13.21 v5.3：高级模型结构

可以逐步尝试：

| 结构 | 用途 |
| --- | --- |
| MLP baseline | 快速验证特征和动作空间 |
| CNN / ResNet | 处理 34 种牌多通道特征 |
| Transformer encoder | 处理牌河、动作序列、回合历史 |
| Graph-like player encoder | 建模四家/三家相对关系 |
| Recurrent memory | 建模长期对局上下文 |
| Mixture of Experts | 区分进攻、防守、染手、三麻等策略专家 |

建议不要一开始就上最大模型。

推荐顺序：

1. MLP baseline。
2. ResNet-style tile encoder。
3. 加入 action history encoder。
4. 加入 multi-head value。
5. 加入 Transformer。
6. 最后再考虑专家模型。

验收标准：

- 每换一个结构都要有消融实验。
- 强度提升必须超过推理成本增加。
- 不追求参数量，优先追求稳定和可部署。

### 13.22 v5.4：强化学习和自对弈规模化

目标：让模型不只模仿历史，而是通过自对弈发现更优策略。

关键模块：

- 并行环境 runner。
- 批量推理。
- 经验回放。
- 策略更新。
- 模型池。
- Elo / 平均顺位评级。
- 对手采样。
- 训练崩坏检测。

注意问题：

- 麻将奖励稀疏。
- 隐藏信息多。
- 多人非零和。
- 三麻和四麻策略差异大。
- 自对弈容易产生奇怪局部策略。

缓解方式：

- 保留规则型 AI 和历史模型作为对手。
- 训练时混入监督学习 loss。
- 保留放铳率、和牌率等约束指标。
- 使用固定评测集发现策略退化。

验收标准：

- 自对弈模型长期强于纯监督模型。
- 模型没有学出明显反人类的坏习惯。
- 训练过程可监控、可中止、可回滚。

### 13.23 v5.5：AI 教练深度学习版

目标：把模型能力转化为玩家能理解的复盘和训练建议。

功能：

- 打牌时显示模型前三推荐。
- 回放时标出关键分歧手。
- 每局生成“最大损失动作”列表。
- 对每个关键动作给出简短解释。
- 显示模型预测的和牌率、放铳率、顺位收益变化。
- 支持对比“你打的牌”和“模型推荐牌”。

解释方式：

- 不要假装模型真的会说人话。
- 解释应结合规则型特征和模型输出。
- 例如：`模型更倾向打 9索，因为保留两面变化，同时该牌对立直家风险较低。`

验收标准：

- 玩家能通过回放知道自己哪里亏。
- 教练建议短而清楚。
- 不因为模型输出不稳定而误导玩家。

### 13.25 硬件和成本预期

不同阶段对硬件要求差异很大。

| 阶段 | 硬件建议 | 说明 |
| --- | --- | --- |
| 数据闭环 | 普通 CPU | 主要是导出和清洗数据 |
| 小模型监督学习 | 单张消费级 GPU 或 CPU 慢训 | 可先验证流程 |
| 中型模型 | 8GB-16GB 显存 GPU | 适合百万级动作样本 |
| 大规模自对弈 | 多进程 CPU + GPU 推理 | 重点是环境吞吐 |
| 大模型训练 | 多 GPU | 只有数据量和评测体系成熟后才值得 |

本项目建议先把小模型跑通，不要一开始追求大模型。

### 13.26 风险清单

深度学习路线的主要风险：

- 数据不干净，模型学到错误规则。
- 样本泄露他家暗手牌，导致模型作弊。
- 训练指标好看，但实战很弱。
- 模型推理太慢，影响网页体验。
- 自对弈策略崩坏。
- 数据集和模型版本不可追溯。
- 没有足够评测局数就误判模型强度。
- 对外部数据使用不合规。

应对策略：

- 所有训练样本必须通过合法性校验。
- 所有 observation 必须做可见性过滤。
- 所有模型上线前必须跑固定评测。
- 所有模型保留回滚路径。
- 所有数据来源写入 `references.md` 或数据集 manifest。

### 14.1 v6.1：多模型生态和模型治理

目标：不再只有一个“当前最强模型”，而是形成多个模型角色、多个模型版本和明确上线流程。

模型类型建议：

| 模型类型 | 作用 |
| --- | --- |
| `fast_policy` | 快速出牌模型，低延迟，适合普通对局 |
| `deep_policy` | 深度策略模型，结合搜索，适合高难度 AI |
| `value_model` | 局面价值模型，预测顺位、点数和局面优劣 |
| `risk_model` | 放铳风险模型，专门评估危险牌 |
| `call_model` | 鸣牌模型，专门判断吃碰杠 |
| `riichi_model` | 立直/默听/弃和模型 |
| `coach_model` | 教练解释模型，结合规则特征生成说明 |
| `opponent_model` | 对手建模模型，预测对手路线和威胁 |

模型治理功能：

- 模型注册表记录每个模型的训练数据、特征版本、动作空间版本和评测报告。
- 模型上线前必须通过规则审计、固定对战评测和性能测试。
- 模型可以灰度发布，只给部分本地配置或测试房间使用。
- 模型可以快速回滚。
- 模型之间可以 A/B 测试。
- 每次模型推理都记录模型版本、输入摘要、输出摘要、耗时和最终动作。

建议新增目录：

```text
model_registry/
├─ models/
│  ├─ fast_policy/
│  ├─ deep_policy/
│  ├─ value_model/
│  └─ coach_model/
├─ manifests/
├─ evaluation_reports/
└─ rollout_configs/
```

验收标准：

- 系统能同时管理多个模型。
- 前端可以选择 AI 风格和模型强度。
- 任一模型推理失败都不会影响规则型 AI 兜底。
- 每个模型都能追溯数据、代码、指标和上线时间。

### 14.2 v6.2：云端训练和分布式自对弈

目标：把训练能力从单机脚本升级为可扩展的批量训练和自对弈系统。

核心能力：

- 训练任务队列。
- 自对弈任务队列。
- 多进程对局生成。
- 多 GPU 训练。
- 训练 checkpoint 自动保存。
- 失败任务自动重试。
- 训练日志统一收集。
- 训练完成自动评测。
- 评测通过后进入候选模型池。

建议拆分服务：

| 服务 | 作用 |
| --- | --- |
| `selfplay_worker` | 批量生成自对弈牌谱 |
| `dataset_builder` | 把原始牌谱转成训练样本 |
| `trainer_worker` | 执行模型训练 |
| `evaluator_worker` | 运行模型对战评测 |
| `model_registry_api` | 管理模型版本和指标 |
| `scheduler` | 调度训练、自对弈和评测任务 |

任务元数据需要记录：

- 任务 ID。
- Git commit。
- 数据集版本。
- 模型配置。
- 训练参数。
- 硬件环境。
- 开始时间。
- 结束时间。
- 输出模型路径。
- 评测报告路径。

验收标准：

- 可以无人值守生成大量自对弈数据。
- 可以排队训练多个模型实验。
- 训练中断后可以从 checkpoint 恢复。
- 每个模型训练结果都有完整报告。

### 14.3 v6.3：统一牌谱协议和数据标准

目标：建立本项目自己的开放牌谱格式，方便导入、导出、训练、回放、测试和第三方工具接入。

建议定义：

- `MahjongGameLog`：整局牌谱。
- `MahjongRoundLog`：单局牌谱。
- `MahjongStepLog`：单步动作。
- `MahjongObservation`：某家可见状态。
- `MahjongAction`：标准动作编码。
- `MahjongResult`：结算结果。
- `MahjongTrainingSample`：训练样本。

协议需要明确：

- 三麻和四麻如何标记。
- 规则档位如何标记。
- 赤宝牌如何标记。
- 拔北如何标记。
- 古役开关如何标记。
- 各家可见信息如何过滤。
- 动作空间版本如何记录。
- 随机种子如何记录。
- 模型评测结果如何绑定牌谱。

格式建议：

| 格式 | 用途 |
| --- | --- |
| `.jsonl` | 调试、回放、逐行读取 |
| `.parquet` | 大规模训练样本 |
| `.mjlog.json` | 本项目完整牌谱 |
| `.sample.npz` | 训练特征张量 |

验收标准：

- 任意一局对局都可以导出成标准牌谱。
- 导出的牌谱可以完整导入并回放。
- 训练样本可以追溯到原始牌谱。
- 协议版本升级时有迁移脚本。

### 14.4 v6.4：AI 对战联赛和评级系统

目标：给不同 AI 版本建立可量化的强度排名，而不是只凭感觉判断强弱。

建议建立内部 AI 联赛：

- 每个模型版本都作为一个参赛 AI。
- 固定规则进行循环赛。
- 分四麻东风、四麻半庄、三麻东风、三麻半庄赛道。
- 每个模型和其他模型打足够多局。
- 计算平均顺位、一位率、放铳率、和牌率、平均打点。
- 建立 Elo 或 Glicko 风格评级。

评测注意：

- 起始点、规则、随机种子必须可复现。
- 要防止单个种子导致偶然波动。
- 每次评测要包含旧模型，避免新模型局部退化。
- 模型上线不只看平均顺位，也看放铳率、卡局率、耗时和非法动作率。

前端展示：

- AI 版本排行榜。
- 模型强度曲线。
- 不同模式下的强弱差异。
- 模型风格标签，例如“防守型”“速攻型”“门清型”“三麻特化”。

验收标准：

- 每个上线模型都有明确评级。
- 新模型必须击败当前基准模型才能升级为默认 L5。
- 模型强弱判断可以复现。

### 14.5 v6.5：个性化 AI 教练

目标：让 AI 教练不只是给通用建议，而是能长期理解某个玩家的习惯和弱点。

个性化数据：

- 玩家常见放铳牌型。
- 玩家常见副露失误。
- 玩家立直过早或过晚的倾向。
- 玩家押退判断偏进攻还是偏保守。
- 玩家在南场、亲家、末位、领先时的决策变化。
- 玩家三麻和四麻表现差异。

个性化教练功能：

- 每局结束给出 3 个最重要复盘点。
- 每周或每 20 局生成一次成长报告。
- 标出玩家最常犯的 5 类错误。
- 给出专项练习建议，例如“立直判断训练”“副露判断训练”“现物防守训练”。
- 根据玩家段位或水平调整提示深度。

训练模式：

- 防守训练。
- 鸣牌训练。
- 立直判断训练。
- 三麻拔北训练。
- 末局点差训练。
- 染手判断训练。
- 役牌速攻训练。

验收标准：

- 教练建议和玩家历史数据有关。
- 复盘不是泛泛而谈，而是引用具体局面。
- 玩家能看到自己的指标变化趋势。

### 14.7 v6.7：开放插件和主题生态

目标：让系统支持可扩展内容，但不影响核心规则安全。

可开放插件：

- 桌布主题。

### 14.8 v6.8：隐私保护和联邦学习方向

目标：如果未来多个用户愿意贡献本地牌谱，系统可以在保护隐私的前提下收集统计或训练信号。

基础原则：

- 默认不上传任何牌谱。
- 用户必须明确开启数据贡献。
- 上传前清除昵称、ID、本地路径等个人信息。
- 用户可以随时关闭并删除贡献数据。
- 数据集 manifest 必须记录来源和许可。

可选技术方向：

- 本地数据匿名化。
- 差分隐私统计。
- 联邦学习实验。
- 只上传梯度或模型更新。
- 只上传聚合指标。

现实建议：

- 短期先做本地匿名导出。
- 中期做自愿上传到私有仓库或服务器。
- 长期再研究联邦学习。

验收标准：

- 用户知道上传了什么。
- 用户可以删除上传数据。
- 训练数据不包含可识别个人信息。

### 14.11 v7.1：高级搜索和不完全信息推理

目标：进一步解决麻将隐藏信息带来的推理问题。

可研究方向：

- 根据可见牌采样对手可能手牌。
- 根据对手弃牌和副露推断手牌分布。
- 根据立直宣言和巡目推断危险度。
- 在多个隐藏状态样本上评估候选动作。
- 使用价值网络评估各个隐藏状态。
- 合并得到最终动作价值。

建议搜索结构：

```text
真实可见状态
→ 采样 N 个可能隐藏状态
→ 每个隐藏状态展开候选动作
→ 模型评估局面价值
→ 聚合期望收益和风险
→ 输出最终动作
```

性能控制：

- 普通对局只采样少量隐藏状态。
- 教练模式可以深度采样。
- 回放离线分析可以使用更多计算。

验收标准：

- 防守判断明显提升。
- 对副露手、立直手、染手的风险估计更合理。
- 推理时间仍可接受。

### 14.13 v7.3：AI 角色和风格系统

目标：让 AI 不只是强弱不同，还能有不同打法风格。

可设计原创 AI 风格：

| 风格 | 特点 |
| --- | --- |
| 守备型 | 放铳率低，领先时更稳 |
| 速攻型 | 副露率高，重视速度 |
| 门清型 | 喜欢立直和高打点 |
| 染手型 | 更愿意做混一/清一 |
| 三麻特化型 | 更懂拔北和三麻点差 |
| 末局追分型 | 末局会强烈按条件进攻 |
| 教练型 | 决策更稳，解释更详细 |

实现方式：

- 同一模型加不同策略温度和权重。
- 训练不同风格数据集。
- 使用条件向量控制风格。
- 在搜索阶段加入风格偏置。

验收标准：

- 玩家能感受到不同 AI 的打法差异。
- 风格差异不等于乱打。
- 每个风格都有统计指标支持。

### 14.14 v7.4：模型解释和可视化研究

目标：让深度学习 AI 的判断更透明。

可视化内容：

- 各候选动作概率。
- 各候选动作价值。
- 和牌率变化。
- 放铳率变化。
- 顺位收益变化。
- 关键可见牌对风险的影响。
- 对手威胁雷达图。
- 当前模型和规则型 AI 分歧。

解释方法：

- 特征重要性分析。
- 候选动作对比。
- 反事实分析，例如“如果打 9索而不是中，会发生什么”。
- 同一局面多模型对比。

验收标准：

- 玩家能理解模型为什么偏向某个动作。
- 开发者能用可视化定位模型问题。
- 解释不会泄露隐藏信息。
